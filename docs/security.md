# VitaCore Security Notes

This document describes the current security and operations behavior implemented in VitaCore. It is documentation only and does not change runtime policy.

## Authentication

VitaCore uses JWT-based authentication for protected backend API routes.

- User passwords are hashed with bcrypt before storage.
- Successful login returns a JWT.
- Protected requests use the `Authorization: Bearer <token>` header.
- Backend authentication middleware verifies the token and attaches the authenticated user context.
- Service logic performs ownership checks before returning or mutating private character, device, measurement, or account data.

JWT secrets must be configured through environment variables. Production secrets should be strong, private, and rotated according to the deployment policy.

## Email Verification

Email verification is used for signup and account recovery flows.

- Verification codes are generated by the backend.
- Codes are hashed before being stored in the database.
- Email delivery is handled through the configured email service.
- Signup requires the email verification flow before account creation is completed.
- Recovery flows validate the submitted code before allowing password reset behavior.

Raw verification codes should not be logged or stored in plaintext.

## Device Verification

Device submission is separated from normal browser interaction.

- Connection codes are used to pair a device with a character.
- Device identifiers are checked before accepting device-submitted measurements.
- Rejected or inactive device behavior is recorded through security events.

This keeps device measurement ingestion behind backend validation instead of trusting client-submitted character ownership alone.

## Security Event Logging

The `security_events` table records important audit and operational events. Its purpose is to preserve a reviewable trail for authentication, device, measurement, command, and anomaly-related activity.

Examples include:

- Login failure.
- Email verification success.
- Device connection or rejection.
- Manual vital update.
- Character command application.
- Anomaly detection.

Older security events may be archived into `security_event_archives`.

## Measurement Anomaly Logging

VitaCore performs lightweight threshold-based anomaly detection for submitted or derived vital values. When a measurement crosses the configured thresholds, the backend records an `ANOMALY_DETECTED` security event.

This is an educational simulation signal, not a clinical diagnosis or medical alerting system.

## CORS Policy

The current development server configuration uses open CORS through Express middleware. This is convenient for local development, but production deployments should restrict allowed origins to the deployed frontend domain or another explicitly approved origin list.

This document does not change the CORS code.

## Startup and Operations Notes

Current startup behavior may not stop the server for every infrastructure problem:

- The database connection check logs connection failure information during startup.
- The security event archive table check can fail and be logged without necessarily blocking server startup.

Production deployments should add or enforce:

- Required environment variable validation before startup.
- Log monitoring for database connection failures.
- Log monitoring for archive table check failures.
- Alerting for repeated authentication failures, device rejections, and anomaly spikes.
- A deployment-specific CORS origin allowlist.

## Sensitive Data Handling

The application should not log or expose:

- Plaintext passwords.
- JWT secrets or issued JWTs.
- Raw email verification codes.
- Raw connection codes beyond the flow that requires them.
- Database credentials.

Local `.env` files should remain outside version control.

## Disclaimer

VitaCore is an educational project. Security controls are implemented to demonstrate responsible application structure, but a production deployment would require a full security review, operational monitoring, secret management, backup strategy, and incident response process.
